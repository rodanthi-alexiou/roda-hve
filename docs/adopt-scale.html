Organizational adoption

Adopt and scale HVE across teams

One developer piloting the loop is useful. An entire organization sharing skills, prompts, and validated IaC is transformational. This page shows how to get from one to the other — told through a fictional partner who did exactly that.

A realistic adoption story

How Contoso Corp scaled from one team to organization-wide

Contoso Corp is a fictional ISV with 12 engineering teams building SaaS products on Azure. One platform team discovered HVE during a hackathon. Six months later, every team uses shared skills and ships with the same RPI discipline. Here is what each phase looked like.

1

Phase 1: Pilot

One team · one project · two weeks

The platform team picks a single greenfield feature — an internal knowledge-base agent — and builds it end-to-end with HVE. The goal is not to ship a product; it is to prove the loop works and measure the delta.

What they did

  • Installed the HVE Core extension in VS Code
  • Configured the CAIRA skill and Microsoft Learn MCP
  • Ran the full RPI loop on their feature: research, plan, implement, review
  • Measured: iterations to working code, bugs caught by the review phase, time to first deploy
What Contoso actually ran

Pilot outcomes

40% fewer iterations

Research-first meant the agent did not invent APIs. The team stopped fixing hallucinated code and started reviewing real code.

Self-heal caught 3 bugs

The review phase caught and fixed issues before any human saw them. No separate QA pass needed for those.

First deploy in 4 days

Using CAIRA IaC instead of authoring Terraform from scratch cut the infra setup from days to an hour.

Pilot success criteria

The pilot proves three things: (1) the loop produces auditable artifacts, (2) the research phase eliminates hallucination, (3) the team can ship faster with less rework. If all three hold, proceed to Phase 2.

2

Phase 2: Establish

Three teams · shared skill library · one month

The pilot team packages their learnings into shared, reusable assets. Two more teams adopt HVE using those assets instead of starting from scratch.

What they built

  • A shared skill library in an internal GitHub repo with domain-specific CAIRA extensions
  • A team playbook: which RPI phases to always run, when to invoke the System Architecture Reviewer, ADR templates
  • A starter template repo with pre-configured .vscode/mcp.json, .copilot/, and azure.yaml
  • Onboarding docs: a 30-minute getting-started guide for new team members

Phase 2 succeeds when a new team can go from zero to first RPI-driven feature in under a day using the shared assets — without needing the pilot team to hold their hand.

3

Phase 3: Scale

All 12 teams · centralized governance · ongoing

HVE is now the default engineering workflow. A platform engineering team maintains the shared assets, measures adoption, and evolves the organization's prompt and skill library.

What they operate

  • A centralized skill repo with versioned releases and a changelog
  • Governance policies: which models are approved, which MCP servers are allowed, security review gates
  • Metrics dashboards: RPI completion rates, self-heal success, time-to-deploy per team
  • Inner-source contribution model: any team can propose new skills or instructions via PR
Contoso's governance model 
# .github/copilot-instructions.md (org-wide)
# Applied to every repo in the Contoso GitHub org

## Security baseline
- All Azure resources use Managed Identity. No connection strings in code.
- All IaC must pass `tfsec` and `checkov` before merge.
- ADRs required for any new external dependency.

## Approved MCP servers
- Microsoft Learn MCP (https://learn.microsoft.com/api/mcp)
- Azure MCP (internal instance)

## Required phases
- Research phase is mandatory for any new service or API integration.
- Review phase runs automatically in CI via the RPI Validator.

Each asset is small on its own. The power is in the combination: a new team member clones the starter repo and immediately has coding standards, security baselines, IaC patterns, and the RPI loop configured — without reading a wiki or attending a training session.
